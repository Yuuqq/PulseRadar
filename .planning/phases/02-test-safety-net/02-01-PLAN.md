<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| PyPI → dev install | New dev deps (pytest-cov, responses, pytest-randomly) enter the development environment from the public index |
| pyproject.toml → CI/local pytest | TOML config drives pytest behavior, including which packages are measured and the fail threshold |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-02-01-01 | Tampering | pyproject.toml coverage source list | mitigate | Acceptance criterion greps for exact `source = ["trendradar", "mcp_server"]`; Pitfall 15 (omitting mcp_server) becomes an explicit test failure. |
| T-02-01-02 | Denial of Service | Dev dep install (PyPI supply chain) | accept | Development-only deps; locked version ranges (`pytest-cov>=7.0,<8` etc.) limit blast radius; no production install path affected. No secrets are added by these deps. |
| T-02-01-03 | Elevation of Privilege | Test infrastructure in production container | mitigate | Dev deps live ONLY in `[dependency-groups] dev` — NOT in `[project.dependencies]`. `pip install trendradar` (production) does not pull them. Acceptance criterion verifies the group location via tomllib parse. |
| T-02-01-04 | Information Disclosure | coverage.xml containing file paths | accept | File paths are repo-relative; no secrets or PII. File is committed ONCE in Plan 06 as documented baseline, then gitignored — not continuously leaked. |
| T-02-01-05 | Tampering | requirements-dev.txt drift from pyproject.toml | mitigate | File is regenerated from pyproject.toml via `uv pip compile` — not hand-edited. Header comment states the generating command so reviewers can regenerate to verify. |
</threat_model>
