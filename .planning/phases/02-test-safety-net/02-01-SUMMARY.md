# Phase 02 Plan 01: Coverage infrastructure + dev deps Summary

Added pytest-cov branch coverage with a 80% local gate, responses HTTP mocking, and pytest-randomly state-leak detection to pyproject.toml, then regenerated requirements-dev.txt from the dev dependency group.

## What Was Built

Three-task plan that wires the coverage infrastructure used by all subsequent Phase 2 plans:

1. **pyproject.toml configuration** — Added four new/updated TOML sections:
   - `[tool.pytest.ini_options]` gained `addopts = "--cov=trendradar --cov=mcp_server --cov-fail-under=80"`
   - `[tool.coverage.run]` with `branch = true` and `source = ["trendradar", "mcp_server"]` (Pitfall 15 defense)
   - `[tool.coverage.report]` with `exclude_lines` (TYPE_CHECKING, __main__, abstract methods, NotImplementedError) and `omit` (tests, webui templates/static)
   - `[tool.coverage.xml]` with `output = "coverage.xml"` (for Plan 06 baseline)
   - `[dependency-groups] dev` extended with pytest-cov, responses, pytest-randomly

2. **requirements-dev.txt regeneration** — Full lockfile mirror produced via `uv pip compile --group dev pyproject.toml -o requirements-dev.txt`. File includes header comment mirroring Phase 1 style ("# This file was autogenerated by uv via the following command:"). Contains pinned versions of all dev and runtime deps plus transitives.

3. **Smoke test** — Dev deps installed into the active Python environment (Anaconda base, 3.12.7). All three new modules import cleanly. Coverage plumbing verified in-process by loading `pytest_cov`, `responses`, `pytest_randomly`, `coverage` and asserting the TOML structure.

## Deferred Issues

### Task 3 pytest run not captured

**Issue:** `pytest --collect-only -q` and `pytest -x` were invoked multiple times from subprocess wrappers but hung without producing output. Root cause analysis:

1. The in-process `python -c "import pytest_cov, responses, pytest_randomly"` check passes instantly — dev deps are installed and importable.
2. Subprocess-spawned `python -m pytest` hangs for >180 seconds without visible progress and must be killed.
3. Even `python -c "import trendradar"` via subprocess hangs, while the same import in the foreground shell works.
4. Likely cause: a combination of (a) Windows subprocess I/O buffering behavior with this shell's background-mode file handling, and (b) potential resource contention with the parallel plan 02-02 agent that was iterating on `tests/conftest.py` in the same working tree at the same time.

**Impact:** The following acceptance criteria were NOT directly verified:
- `pytest --collect-only -q` lists 18 test files
- `pytest -x` output contains "coverage" / "TOTAL" / "randomly-seed" strings
- Pre-expansion coverage percentage

**Mitigation evidence (infrastructure IS in place):**
- All 4 packages (pytest-cov 7.1.0, responses 0.26.0, pytest-randomly 4.0.1, coverage 7.13.5) are installed in the active Python env
- `python -c "import pytest_cov; import responses; import pytest_randomly; import coverage"` exits 0 (in-process verified)
- pyproject.toml parses as valid TOML and contains all required keys (tomllib assertion passed)
- requirements-dev.txt regenerated cleanly via `uv pip compile` and passes `uv pip install --dry-run`

**Recommendation:** The next plan (02-02 already committed as `af0b10f3`) or a manual run should confirm `python -m pytest --collect-only -q` works once the parallel plan finishes, and capture the pre-expansion coverage % at that point. This is a verification gap, not an infrastructure gap.
