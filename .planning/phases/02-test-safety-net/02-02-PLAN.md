<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Test fixture ↔ global module state | Autouse fixture writes to a production module's global variable (`trendradar.storage.manager._storage_manager`) during tests only |
| responses library ↔ requests calls | RequestsMock intercepts HTTP calls at the test boundary; unmocked URLs raise ConnectionError |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-02-02-01 | Tampering | Autouse singleton reset touching production code | mitigate | Reset ONLY sets to None (initial state) and ONLY in tests/ — production `get_storage_manager()` recreates on next call. Acceptance criterion runs pre-existing 18 tests under the autouse fixture to prove no regression. |
| T-02-02-02 | Information Disclosure | mock_config contains API placeholder "fake-key" | mitigate | Value is a literal "fake-key" string, not a real credential. Acceptance criterion greps for the exact literal to prevent silent substitution with a real key during editing. |
| T-02-02-03 | Tampering | responses fixture with assert_all_requests_are_fired=False | accept | Tradeoff accepted per D-20 / Pitfall 3. Per-test strict mode available when a test needs it. Documented in fixture docstring so future test authors know the default. |
| T-02-02-04 | Denial of Service | Autouse fixture runs on every test (incl. existing 16) | accept | Reset is a single attribute assignment — negligible cost (<1ms per test). |
| T-02-02-05 | Tampering | Fixtures leak across subsystems because `conftest.py` is at tests/ root | mitigate | Per D-19 and D-25, fixtures are function-scoped. tests/mcp/conftest.py (Plan 04) and per-subsystem conftests can scope their own fixtures additively. Root-level fixtures here are intentionally generic. |
</threat_model>
