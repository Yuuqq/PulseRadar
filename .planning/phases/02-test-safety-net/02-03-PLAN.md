    ```python
    from __future__ import annotations

    from trendradar.crawler.base import CrawlResult, FetchedItem


    def assert_fetched_item_shape(item: FetchedItem) -> None:
        """Assert a FetchedItem has the expected field types and non-empty title."""
        assert isinstance(item, FetchedItem), f"not a FetchedItem: {type(item)!r}"
        assert isinstance(item.title, str) and len(item.title) > 0, "title must be non-empty str"
        assert isinstance(item.url, str), "url must be str"
        assert isinstance(item.rank, int), "rank must be int"


    def assert_crawl_result_success(result: CrawlResult, min_items: int = 1) -> None:
        """Assert a CrawlResult represents a successful crawl with ≥ min_items items."""
        assert result.success, f"expected success but got errors: {result.errors}"
        assert len(result.items) >= min_items, (
            f"expected ≥{min_items} items, got {len(result.items)}"
        )
        for item in result.items:
            assert_fetched_item_shape(item)


    def assert_crawl_result_error(result: CrawlResult) -> None:
        """Assert a CrawlResult represents a failed crawl (no items, ≥1 error msg)."""
        assert not result.success, "expected failure but crawl was successful"
        assert len(result.errors) > 0, "expected ≥1 error message"
        assert len(result.items) == 0, f"expected empty items on failure, got {len(result.items)}"
    ```

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Test fixtures (inline HTML/JSON) → plugin parsers | Crawler plugins parse untrusted third-party JSON/HTML; tests register their own inline fixtures as if from upstream |
| responses mock → requests.Session.get | `responses` patches the requests library at import time; failing to register a URL causes ConnectionError (safer than silent passthrough) |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-02-03-01 | Tampering | Test fixtures with malformed JSON/HTML parsed by production plugins | accept | This is the POINT of error-mode tests — fixtures represent untrusted input. Plugin parsers must handle malformed input gracefully (plugin contract: never raise). Tests verify this contract. |
| T-02-03-02 | Information Disclosure | Real API endpoints leaking in URL strings in test files | accept | URLs are public documentation values; no secrets or auth tokens appear in test files. The `api_key` fields contain the literal `"fake-key"`. |
| T-02-03-03 | Denial of Service | Test suite making real HTTP calls if mock not registered | mitigate | `@responses.activate` raises ConnectionError on unmocked URLs by design. Acceptance criterion verifies tests pass (no ConnectionError). |
| T-02-03-04 | Spoofing | Plugin could silently accept wrong-shape response | mitigate | Each happy-path test asserts concrete field values flow through (e.g., title substring) — not just `result.success`. Pitfall 4 anti-coverage-padding satisfied. |
| T-02-03-05 | Tampering | EastMoney/TongHuaShun regex parsing diverging between test fixture and real traffic | accept | Pitfall 1 policy: lock current behavior, not ideal. If a plugin fixture shape differs from live traffic, that's a separate concern for a future v2 "plugin live smoke test" requirement. |
</threat_model>
