<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| coverage.xml content -> git history | coverage.xml contains file paths and line-level coverage data; committed once as a baseline reference |
| pytest-cov -> pyproject.toml config | Coverage config in pyproject.toml drives what is measured and the fail threshold |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-02-06-01 | Information Disclosure | coverage.xml containing repo file paths in git history | accept | File paths are repo-relative (e.g., `trendradar/core/pipeline.py`); no secrets, credentials, or PII. Committed once as documented baseline, then gitignored. |
| T-02-06-02 | Tampering | coverage.xml manually edited to inflate baseline | mitigate | Baseline is generated by `pytest --cov` in the same commit session as the test results. Reviewers can regenerate by running `pytest` to verify. The per-module table in SUMMARY.md provides a human-readable cross-check. |
| T-02-06-03 | Denial of Service | .gitignore change accidentally excluding other files | mitigate | Only the literal line `coverage.xml` is appended. Acceptance criterion verifies existing entries (e.g., `__pycache__/`) are preserved. No glob patterns used. |
| T-02-06-04 | Tampering | coverage gate lowered to make baseline pass | mitigate | The `--cov-fail-under=80` in pyproject.toml was set in Plan 01 and is NOT modified by this plan. Acceptance criterion verifies pytest exits 0 without any `--cov-fail-under` override. |
</threat_model>
