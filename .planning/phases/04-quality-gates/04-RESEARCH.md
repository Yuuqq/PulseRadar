## Summary

Phase 4 establishes automated quality gates to prevent regression in the newly-refactored TrendRadar codebase. The phase configures Ruff (a fast Python linter/formatter written in Rust) with specific rule sets (I, F, E, UP, B, SIM, RUF) and integrates pre-commit hooks to enforce checks on every commit in under 10 seconds.

**Current state analysis:**

- Pre-commit 4.5.0 is already installed globally
- No existing Ruff configuration in pyproject.toml
- No .pre-commit-config.yaml file exists
- Codebase has ~119 Python files (trendradar + mcp_server)
- Running `ruff check . --select I,F,E,UP,B,SIM,RUF` exits clean (0 violations)
- Running `ruff format --check .` shows ~138 files would be reformatted

**Primary recommendation:** Add Ruff configuration to pyproject.toml with the specified rule sets, run `ruff format .` to format the codebase, create .pre-commit-config.yaml with ruff-pre-commit hooks, and verify all checks pass before committing.

## Architecture Patterns

### Recommended Configuration Structure
```
project_root/
├── pyproject.toml           # Ruff config in [tool.ruff] section
├── .pre-commit-config.yaml  # Pre-commit hook definitions
└── .git/hooks/              # Auto-generated by pre-commit install
```

### Pattern 1: Ruff Configuration in pyproject.toml
**What:** Centralized linting and formatting configuration
**When to use:** Always - keeps all Python tool config in one place
**Example:**
```toml
# Source: Ruff official docs - https://docs.astral.sh/ruff/configuration/
[tool.ruff]
target-version = "py310"
line-length = 100

[tool.ruff.lint]
select = [
    "I",    # isort - import sorting
    "F",    # Pyflakes - basic errors
    "E",    # pycodestyle errors
    "UP",   # pyupgrade - modern Python syntax
    "B",    # flake8-bugbear - likely bugs
    "SIM",  # flake8-simplify - simplification suggestions
    "RUF",  # Ruff-specific rules
]
ignore = []

[tool.ruff.lint.isort]
known-first-party = ["trendradar", "mcp_server"]
```

### Pattern 2: Pre-commit Hook Configuration
**What:** Automated checks on git commit
**When to use:** Always - prevents bad code from entering the repo
**Example:**
```yaml
# Source: pre-commit official docs + ruff-pre-commit
repos:
  - repo: https://github.com/astral-sh/ruff-pre-commit
    rev: v0.8.4  # Use latest stable version
    hooks:
      - id: ruff
        args: [--fix]  # Auto-fix issues when possible
      - id: ruff-format
```

### Pattern 3: Incremental Adoption Strategy
**What:** Format and lint in stages to avoid massive diffs
**When to use:** When adding quality gates to existing codebases
**Example:**
```bash
# Stage 1: Format all files (one-time)
ruff format .

# Stage 2: Verify linting passes
ruff check . --select I,F,E,UP,B,SIM,RUF

# Stage 3: Install pre-commit hooks
pre-commit install

# Stage 4: Test hooks work
git add -A
git commit -m "test: verify pre-commit hooks"
```

### Anti-Patterns to Avoid
- **Enabling all rules at once:** Start with core rule sets (I, F, E) then add others incrementally
- **Ignoring formatting differences:** Ruff format is opinionated - accept its style rather than fighting it
- **Skipping pre-commit install:** Hooks only work after `pre-commit install` is run
- **Using --no-verify to bypass hooks:** Defeats the purpose of quality gates

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Import sorting | Custom import organization script | Ruff (I rules) | Handles stdlib/third-party/first-party detection, respects __future__ imports |
| Code formatting | Manual style guide enforcement | Ruff format | Deterministic, fast, handles edge cases (long strings, nested structures) |
| Git hook management | Shell scripts in .git/hooks/ | pre-commit framework | Cross-platform, version-controlled, supports multiple languages |
| Lint rule configuration | Multiple tool configs (flake8, black, isort) | Single pyproject.toml [tool.ruff] | One source of truth, no config conflicts |

**Key insight:** Ruff consolidates 10+ Python tools into one fast binary. Custom solutions can't match its performance (written in Rust) or comprehensive rule coverage (700+ rules available).

## Common Pitfalls

### Pitfall 1: Formatting Conflicts with Existing Code Style
**What goes wrong:** Team members reject auto-formatting because it changes "their" style
**Why it happens:** Ruff format is opinionated and non-configurable (by design)
**How to avoid:** Format entire codebase in one commit with clear commit message explaining the change is automated
**Warning signs:** PRs with formatting-only changes mixed with logic changes

### Pitfall 2: Pre-commit Hooks Not Installed
**What goes wrong:** Developers commit code that fails CI because hooks weren't run locally
**Why it happens:** `pre-commit install` is a manual step after cloning
**How to avoid:** Add installation check to README, include in onboarding docs, consider adding to setup script
**Warning signs:** CI failures on commits that should have been caught locally

### Pitfall 3: Slow Hook Execution
**What goes wrong:** Pre-commit hooks take >10 seconds, developers use --no-verify
**Why it happens:** Running checks on all files instead of just staged files
**How to avoid:** Use default pre-commit behavior (only checks staged files), avoid `--all-files` in hooks
**Warning signs:** Developers complaining about slow commits, frequent --no-verify usage

### Pitfall 4: Ignoring Ruff's Auto-fixes
**What goes wrong:** Manually fixing issues that Ruff can fix automatically
**Why it happens:** Not using `--fix` flag or `ruff-format` hook
**How to avoid:** Always use `ruff check --fix` and include `args: [--fix]` in pre-commit config
**Warning signs:** Repetitive manual edits for import sorting or simple style issues

### Pitfall 5: Version Mismatch Between Local and CI
**What goes wrong:** Code passes locally but fails in CI due to different Ruff versions
**Why it happens:** Using globally installed Ruff instead of project-pinned version
**How to avoid:** Add ruff to dev dependencies in pyproject.toml, use `pre-commit autoupdate` regularly
**Warning signs:** "Works on my machine" syndrome with linting

### Pre-commit Configuration with Ruff
```yaml
# Source: https://github.com/astral-sh/ruff-pre-commit
repos:
  - repo: https://github.com/astral-sh/ruff-pre-commit
    rev: v0.8.4
    hooks:
      - id: ruff
        args: [--fix]
      - id: ruff-format
```

### Running Ruff Checks Manually
```bash
# Check for issues (exit code 1 if violations found)
ruff check .

# Check with auto-fix
ruff check . --fix

# Check specific rule sets
ruff check . --select I,F,E,UP,B,SIM,RUF

# Format code
ruff format .

# Check if code is formatted (exit code 1 if not)
ruff format --check .
```

### Installing and Testing Pre-commit Hooks
```bash
# Install hooks (one-time per clone)
pre-commit install

# Run hooks manually on all files
pre-commit run --all-files

# Run hooks on staged files (what happens on commit)
pre-commit run

# Update hook versions
pre-commit autoupdate
```

### Known Threat Patterns for Quality Tooling

| Pattern | STRIDE | Standard Mitigation |
|---------|--------|---------------------|
| Malicious pre-commit hooks | Tampering | Pin hook versions in .pre-commit-config.yaml, review hook sources |
| Supply chain attack via Ruff | Tampering | Use official Ruff releases, verify checksums, pin versions |
| Bypassing quality gates with --no-verify | Tampering | CI enforcement (run same checks in CI), team policy |

**Note:** The primary security concern is ensuring quality gates cannot be bypassed. This is addressed by:
1. Running identical checks in CI (not just locally)
2. Pinning tool versions to prevent unexpected behavior changes
3. Using official, well-maintained tools (Ruff, pre-commit) with strong security track records
